# 实验方案文档

> 对应论文第五章：系统实现与实验评估
> 目标：通过三类实验验证多模态 GraphRAG 工作流生成系统在电力系统场景下的有效性

## 1. 总体实验设计

围绕三个研究问题（RQ）展开：

| 编号 | 研究问题 | 对应实验 |
|------|---------|---------|
| RQ1 | 多模态图谱构建的质量如何？流程图解析是否带来增益？ | 实验一 |
| RQ2 | GraphRAG 检索相比向量 RAG 是否提供更优上下文？ | 实验二 |
| RQ3 | 完整系统生成的工作流相比基线是否更完整、更可执行？ | 实验三 |

每个 RQ 都对应一组消融，用来支撑论文创新点。

## 2. 数据集

课题定位为**电力系统专业场景中的多源异构工作流生成**。数据来源统一收敛到电力系统公开数据集与开源图谱项目，选取 OPS/KQA/DA 三类任务；当前论文实验已从 6 条小样本扩展到 30 条人工金标工作流，后续如时间允许可继续扩展到每类不少于 20 条任务，并保留数据源、抽样规则和金标标注记录。

### 2.1 场景一：运维故障排查（OPS）

- **主来源**：PowerGridQA 的 NERC Operational Insight 子集、OpenEI/OEDI Event-correlated Outage Dataset in America。
- **扩展来源**：Texas A&M Electric Grid Test Cases 中的合成测试网、ENTSO-E Transparency Platform 中的欧洲运行数据。
- **多模态成分**：故障叙述、停电影响指标、合成电网拓扑、排障/恢复流程图。
- **任务示例**："110kV 变电站母线差动保护动作，三相短路故障，给出故障排查和恢复送电工作流"
- **标注**：人工标注 gold workflow（步骤序列 + 智能体分配 + 工具/数据需求）。

### 2.2 场景二：知识库问答（KQA）

- **主来源**：PowerGridQA 的 Power System Theory、NERC、Reasoning 三个子集。
- **扩展来源**：Electricity Knowledge Graph 的 RDF/元数据组织方式、CIM-Graph 的 CIM 模型图结构。
- **多模态成分**：问答文本、设备/概念关系图、CIM 模型示意图、知识图谱子图。
- **任务示例**："根据NERC经验与电力系统理论，说明继电保护误动后的诊断工作流"
- **标注**：将问答目标改写为可执行工作流任务，并标注所需检索、诊断、校验和报告角色。

### 2.3 场景三：数据分析流水线（DA）

- **主来源**：Open Power System Data (OPSD) 时间序列数据，当前本地文件为 `time_series_60min_singleindex.csv`。
- **扩展来源**：ENTSO-E Transparency Platform 的实时/历史欧洲电力系统数据。
- **多模态成分**：CSV schema、时序统计摘要、负荷/新能源曲线图、分析流程图。
- **任务示例**："基于近30天OPSD负荷历史数据，预测未来24小时分时段负荷，识别峰谷差并给出需求侧管理建议"
- **标注**：标注期望步骤（加载→清洗→异常检测→预测/统计→可视化→报告）。

### 2.4 数据集规模与划分

| 场景 | 当前已落地任务 | 改进实验建议 | 主数据源 |
|------|--------------|-------------|---------|
| OPS  | 10 | 20--50 | PowerGridQA-NERC + OpenEI/OEDI + Texas A&M |
| KQA  | 10 | 20--50 | PowerGridQA + Electricity Knowledge Graph + CIM-Graph |
| DA   | 10 | 20--50 | OPSD + ENTSO-E |

> 因为系统不训练模型（只是 prompt + 检索），不设置传统训练集；扩展实验可设置少量验证任务用于调节 `top_k`、`max_hops` 与算法 C 权重，测试任务只用于最终报告。

## 3. 基线方法

| 编号 | 基线 | 描述 | 用途 |
|------|------|------|------|
| B1 | Pure-LLM | 直接 GPT-4o-mini 生成工作流，无检索 | 证明 RAG 必要性 |
| B2 | Vector-RAG | 传统向量 RAG + LLM 生成 | 证明图结构必要性 |
| B3 | Text-GraphRAG | 仅文本 GraphRAG（去掉图像和流程图） | 证明多模态必要性 |
| B4 | GraphRAG-NoMatch | 我们的 GraphRAG，但用 LLM 自由分配角色（去掉智能体匹配算法） | 证明算法 C 必要性 |
| **B5** | **Ours-Full** | 完整系统 | 主方法 |

## 4. 评估指标

### 4.1 图谱构建质量（RQ1 → 实验一）

| 指标 | 定义 |
|------|------|
| 节点 P/R/F1 | vs 人工标注的实体集 |
| 关系 P/R/F1 | vs 人工标注的三元组 |
| 流程图节点准确率 | 流程节点抽取正确率 |
| 跨模态对齐准确率 | 抽样 100 条跨模态边人工判定 |

### 4.2 GraphRAG 检索质量（RQ2 → 实验二）

| 指标 | 定义 |
|------|------|
| Recall@5 / Recall@10 | 命中相关节点比例 |
| MRR | 平均倒数排名 |
| 上下文相关性 | LLM-as-judge 1-5 打分 |
| 上下文压缩率 | 输入 token 数 / 全图 token 数 |

### 4.3 工作流生成质量（RQ3 → 实验三）⭐ 最重要

| 指标 | 定义 | 评估方式 |
|------|------|---------|
| **步骤完整性** | 预测步骤覆盖 gold 步骤的比例 | 自动 + 人工 |
| **逻辑正确性** | 步骤顺序与依赖是否合理 | LLM-as-judge + 人工 |
| **可执行率** | DAG 合法 + IO 兼容 + 工具存在 | 自动 |
| **工具调用准确率** | 工具选择是否正确 | 自动比对 |
| **角色分配准确率** | 智能体分配是否合理 | 人工抽检 |
| Token 成本 | 平均生成一个工作流的 LLM token 数 | 自动 |
| 端到端延时 | 平均响应时间（秒） | 自动 |

**目标**：回答 RQ2。

## 8. 可能的风险与预案

| 风险 | 预案 |
|------|------|
| 数据标注工作量过大 | 用 GPT-4o 做半自动预标注，人工只做修正 |
| 流程图解析准确率低 | 退化为"不解析直接当图像"，作为消融对比 |
| 三场景实验跑不完 | 优先保 OPS（最贴合任务书参考文献），KQA 和 DA 视情况 |
| LLM 评分不稳定 | 同一样本评 3 次取平均，关键指标人工复核 |
